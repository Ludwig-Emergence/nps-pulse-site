## 1. 用户场景与痛点分析

### 1.1 目标用户画像

NPS GO 的目标用户是处于业务 0→1 阶段的个体经营者，通常具有以下共同特征：

- 年营收 < 50 万元，团队 ≤ 3 人（多数为单人运营）
- 依赖微信生态（朋友圈、小程序、社群）获客和交付
- 没有专职客服/运营人员，所有客户关系由主理人亲自维护
- 对"系统化管理"有认知但缺乏工具和方法论

#### 角色 A：个人品牌主理人（知识付费/咨询型）

| 维度 | 描述 |
|------|------|
| **典型身份** | 独立咨询师、个人IP博主、知识付费课程讲师 |
| **代表案例** | 做小红书心理咨询内容的独立咨询师，有200+付费学员 |
| **业务模式** | 通过内容引流 → 社群运营 → 课程/1v1咨询变现 |
| **技术水平** | 会用微信、小红书、飞书等工具，不会写代码 |
| **核心关切** | 学员续费率、口碑转介绍、如何知道哪些学员满意哪些不满意 |
| **月营收** | 1-5 万元 |
| **愿意付费** | 500-3000 元/年（对"能直接帮我赚钱或省钱的工具"敏感） |

#### 角色 B：微型电商商家（实物/本地服务型）

| 维度 | 描述 |
|------|------|
| **典型身份** | 微信小店/有赞/拼多多小型卖家、本地烘焙/花艺工作室主理人 |
| **代表案例** | 在微信私域卖手工烘焙产品的宝妈，月订单300+ |
| **业务模式** | 私域引流 → 微信成交 → 复购驱动增长 |
| **技术水平** | 熟练使用电商平台后台、微信，不会数据分析 |
| **核心关切** | 差评来了才知道问题，无法提前预警；不知道哪些客户可能流失 |
| **月营收** | 2-10 万元 |
| **愿意付费** | 1000-5000 元/年（对"直接减少差评和退货"敏感） |

#### 角色 C：线下实体小老板（服务型门店）

| 维度 | 描述 |
|------|------|
| **典型身份** | 美甲店、宠物店、小型健身房、社区咖啡馆老板 |
| **代表案例** | 开了2年的社区宠物店，有500+老客户微信 |
| **业务模式** | 到店体验 → 加微信 → 老带新裂变 |
| **技术水平** | 会用微信和大众点评后台，对"系统"有畏惧感 |
| **核心关切** | 老客户为什么不来了？怎么让满意客户帮我推荐？ |
| **月营收** | 3-15 万元 |
| **愿意付费** | 500-3000 元/年（对"帮我留住老客户"敏感） |

### 1.2 使用场景

#### 场景 A1：知识付费主理人 — 学员满意度黑箱

**背景：** 张老师卖了一个 299 元的线上课程，有 150 个学员。课程结束后，她想知道学员满意度，但：
- 群里发问卷 → 没人填（回收率 < 10%）
- 私聊问 → 太耗时，而且学员不好意思说真话
- 朋友圈发反馈截图 → 只有好评，幸存者偏差

**使用 NPS GO：**
1. 通过 NPS GO 生成一个 NPS 调研链接/小程序卡片
2. 在课程社群里一键发送，学员 3 秒完成打分（0-10 分 + 可选留言）
3. 系统自动计算 NPS 分值，生成可视化报告
4. 张老师一眼看到：Promoter（9-10 分）占比 40%，Passive（7-8 分）占比 35%，Detractor（0-6 分）占比 25%
5. 对 Detractor 学员的留言逐条分析，发现"课程太理论缺实操"是核心不满
6. 下期课程增加实操环节，NPS 提升 15 分

#### 场景 B1：微型电商 — 差评预警缺失

**背景：** 李姐在微信私域卖手工蛋糕，月销 300 单。某天突然发现退款率飙升，但不知道原因：
- 有赞后台只有退款数据，没有"满意度趋势"
- 微信私聊问客户 → 客户已读不回
- 发现问题时，已经在小红书看到差评帖子了

**使用 NPS GO：**
1. 每月订单完成后，NPS GO 自动发送 NPS 调研（对接订单系统或手动导入订单列表）
2. 系统实时监控：如果某天 Detractor 比例突增 → 自动触发告警
3. 李姐第一时间看到：最近 3 天有 8 个 Detractor，留言集中反映"配送时蛋糕变形"
4. 立即更换包装方案，一周后 Detractor 比例回落

#### 场景 C1：线下门店 — 老客户沉默流失

**背景：** 王老板开了一家社区宠物店，微信上有 500 个老客户。近半年发现：
- 月活客户从 200 降到 120
- 大众点评评分 4.8，看起来没问题
- 但不知道哪些老客户已经不来了，为什么不来

**使用 NPS GO：**
1. 每季度通过 NPS GO 给老客户发送 NPS 调研（微信消息卡片形式）
2. 发现：Promoter 客户平均月消费 350 元，Detractor 客户已停止消费
3. Detractor 留言中，"洗澡等待时间太长"是排名第一的不满
4. 王老板调整预约排期，减少等待时间
5. 对 Detractor 客户发送专属优惠券 → 其中 30% 回流消费

### 1.3 痛点优先级排序

| 优先级 | 痛点 | 影响角色 | 严重程度 | NPS GO 能否解决 |
|--------|------|----------|----------|----------------|
| **P0** | 不知道客户真实满意度，决策靠猜 | A, B, C | 🔴 极高 | ✅ 核心解决 |
| **P0** | 满意度数据收集成本高（问卷没人填） | A, B, C | 🔴 极高 | ✅ 核心解决 |
| **P1** | 差评/不满发现滞后，错过挽回窗口 | B, C | 🟠 高 | ✅ 告警功能解决 |
| **P1** | 不知道哪些客户可能流失 | A, B, C | 🟠 高 | ✅ 客户分层解决 |
| **P2** | 无法系统化追踪满意度变化趋势 | A, B | 🟡 中 | ✅ 报告功能解决 |
| **P2** | 满意客户无法被识别和激活转介绍 | A, C | 🟡 中 | ✅ Promoter 标记解决 |
| **P3** | 不懂 NPS 方法论，不知从何开始 | A, B, C | 🟢 低 | ✅ 引导 + 模板解决 |

## 3. 功能 PRD（产品需求文档）

### 3.1 F1 — NPS 调研创建

#### 功能描述
用户通过 3 步流程快速创建一份 NPS 调研问卷，包含品牌信息、核心 NPS 问题和可选附加问题。

#### 用户故事
- **As a** 个人品牌主理人，**I want** 用最少的步骤创建一份专业的 NPS 问卷，**so that** 我不需要学习复杂工具就能开始收集客户反馈。
- **As a** 微型电商商家，**I want** 在问卷中展示我的品牌 logo 和名称，**so that** 客户知道这是来自我的调研，提升信任感和回复率。

#### 交互流程

**第一步：基本信息设置**
1. 用户点击"创建新调研"
2. 填写调研名称（如"2026年7月课程满意度"）
3. 上传品牌 logo（可选，支持 JPG/PNG，≤ 2MB）
4. 设置品牌主色（默认 #1677FF）

**第二步：问卷内容配置**
1. 核心问题（必填，默认不可删除）：
   - "您在多大程度上愿意向朋友推荐 [品牌名]？"（0-10 分）
2. 附加问题（可选，最多 3 题）：
   - 预设模板选择（如"您最喜欢我们哪一点？""我们最需要改进什么？"）
   - 自定义问题（文本输入）
3. 感谢语配置：
   - 默认："感谢您的反馈！我们会持续改进。"
   - 可自定义

**第三步：预览与发布**
1. 实时预览问卷效果（模拟客户端展示）
2. 确认后点击"发布"
3. 系统生成分发链接/二维码

#### 数据字段/逻辑

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| survey_id | UUID | 自动 | 调研唯一标识 |
| user_id | String | 自动 | 创建者 ID |
| survey_name | String | 是 | 调研名称，≤ 50 字符 |
| brand_logo | URL | 否 | 品牌 logo 图片地址 |
| brand_color | Hex | 是 | 品牌主色，默认 #1677FF |
| core_question | String | 是 | NPS 核心问题文本 |
| additional_questions | Array | 否 | 附加问题列表，最多 3 个 |
| thank_you_message | String | 是 | 感谢语 |
| status | Enum | 自动 | draft / active / closed |
| created_at | DateTime | 自动 | 创建时间 |
| distribution_url | String | 自动 | 分发链接 |

**业务逻辑：**
- 每个用户最多同时有 10 个 active 状态的调研
- 调研创建后默认为 draft，分发后变为 active
- 用户可手动关闭调研（变为 closed），关闭后不再接收新回复

#### 验收标准
1. ✅ 用户能在 3 步内完成调研创建，全程 ≤ 2 分钟
2. ✅ 预览效果与客户端实际展示一致
3. ✅ 支持 logo 上传和品牌色自定义
4. ✅ 附加问题支持预设模板和自定义输入
5. ✅ 发布后自动生成可用的分发链接

### 3.2 F2 — 多渠道分发

#### 功能描述
为已创建的 NPS 调研生成多种分发形式（链接、二维码、微信卡片），支持一键复制/下载/分享，降低客户填写门槛。

#### 用户故事
- **As a** 知识付费主理人，**I want** 在微信社群里一键发送调研卡片，**so that** 学员不需要跳转浏览器就能直接打分，提升回收率。
- **As a** 线下门店老板，**I want** 生成一个二维码放在收银台，**so that** 客户扫码即可评价，不用加微信再发链接。

#### 交互流程

**分发渠道生成：**
1. 用户进入已创建的调研详情页
2. 点击"分发"按钮
3. 展示可选分发渠道：
   - **链接复制**：一键复制 HTTPS 短链接
   - **二维码下载**：生成二维码图片（PNG/SVG），支持自定义尺寸
   - **微信卡片**：生成带品牌 logo 和描述的分享卡片（适用于微信/企业微信）
   - **批量导入发送**：上传客户手机号/微信号列表，系统自动发送（增强功能）

**客户端体验：**
1. 客户点击链接/扫码/点击卡片
2. 进入品牌化的 NPS 问卷页面（展示品牌 logo + 品牌色）
3. 点击 0-10 分（大按钮，单手可操作）
4. 如有附加问题，滑动填写（可跳过）
5. 点击"提交" → 展示感谢语
6. 全程 ≤ 15 秒

#### 数据字段/逻辑

| 字段 | 类型 | 说明 |
|------|------|------|
| distribution_id | UUID | 分发记录唯一标识 |
| survey_id | UUID | 关联的调研 |
| channel | Enum | link / qrcode / wechat_card / batch_sms |
| short_url | String | 短链接 |
| qr_code_url | String | 二维码图片地址 |
| batch_list | Array | 批量发送的客户列表（增强功能） |
| click_count | Integer | 点击/扫码次数 |
| submit_count | Integer | 实际提交数 |

**业务逻辑：**
- 短链接使用 6 位随机字符，有效期跟随调研状态
- 二维码默认 300x300px，支持 500/800/1000px
- 微信卡片格式遵循微信 JSSDK 分享规范
- 客户端页面为 H5 响应式设计，适配手机端
- 同一客户（通过设备指纹 + cookie 识别）同一调研只能提交一次

#### 验收标准
1. ✅ 生成的链接和二维码可在手机端正常访问
2. ✅ 客户端问卷页面品牌化展示正确
3. ✅ 客户从打开到提交 ≤ 15 秒
4. ✅ 同一客户无法重复提交
5. ✅ 二维码可下载为 PNG 和 SVG 格式
6. ✅ 分发数据统计（点击数、提交数）实时更新

### 3.3 F3 — 自动计分与分类

#### 功能描述
对收集到的 NPS 打分数据进行自动计算和分类，将每个客户归入 Promoter（推荐者）、Passive（被动满意者）、Detractor（贬损者）三个类别。

#### 用户故事
- **As a** 微型商家，**I want** 系统自动帮我算出 NPS 分数，**so that** 我不需要理解公式也能知道我的客户满意度水平。

#### 交互流程

1. 客户提交 NPS 打分后，系统实时计算
2. 分类规则自动执行，客户被标记为对应类别
3. 用户在数据看板中查看汇总结果

#### 数据字段/逻辑

**分类规则（标准 NPS 模型）：**

| 分数区间 | 分类 | 标签 | 颜色 |
|----------|------|------|------|
| 9-10 | Promoter | 推荐者 | 🟢 绿色 |
| 7-8 | Passive | 被动满意者 | 🟡 黄色 |
| 0-6 | Detractor | 贬损者 | 🔴 红色 |

**NPS 计算公式：**
```
NPS = (Promoter 数量 - Detractor 数量) / 总回复数 × 100
```

**NPS 范围：** -100 到 +100

**行业参考基准：**
- NPS > 50：优秀
- NPS 30-50：良好
- NPS 0-30：一般
- NPS < 0：需要关注

**每条回复的数据结构：**

| 字段 | 类型 | 说明 |
|------|------|------|
| response_id | UUID | 回复唯一标识 |
| survey_id | UUID | 关联调研 |
| score | Integer | 0-10 的 NPS 分数 |
| category | Enum | promoter / passive / detractor |
| comment | String | 文字留言（可选，≤ 500 字符） |
| customer_info | Object | 客户信息（如有采集） |
| submitted_at | DateTime | 提交时间 |
| device_fingerprint | String | 设备指纹（防重复提交） |

#### 验收标准
1. ✅ 客户提交后 ≤ 1 秒完成分类计算
2. ✅ NPS 分值计算公式正确（边界值测试：全 Promoter = 100，全 Detractor = -100）
3. ✅ 分类结果实时反映在数据看板
4. ✅ 客户端提交成功后看到感谢语（不显示分类结果）

### 3.4 F4 — 数据看板

#### 功能描述
以可视化方式展示 NPS 调研的核心数据指标，包括总 NPS 分值、客户分类占比、回复趋势、留言汇总等。

#### 用户故事
- **As a** 个人品牌主理人，**I want** 打开看板就能一眼看清我的 NPS 整体水平和变化趋势，**so that** 我不用做 Excel 就能做出数据驱动的决策。

#### 交互流程

**看板入口：** 用户登录后首页即为数据看板

**看板内容：**

1. **顶部核心指标卡片**
   - 总 NPS 分值（大数字展示，带颜色标识）

   - 每个 Tab 展示客户列表（如有采集客户信息）

   - 展示所有文字留言
   - 按时间倒序排列
   - 高亮显示 Detractor 留言（红色背景）

### 3.5 F5 — 客户分层标签

#### 功能描述
基于 NPS 打分结果自动给客户打标签，支持标签管理和客户列表导出，帮助用户进行精细化运营。

#### 用户故事
- **As a** 微型电商商家，**I want** 导出 Promoter 客户名单并给他们发专属优惠，**so that** 我能激活高满意度客户的复购和转介绍。

#### 交互流程

1. 系统自动根据 NPS 分数给客户打标签（Promoter / Passive / Detractor）
2. 用户可在"客户管理"页面查看所有带标签的客户
3. 支持按标签筛选客户列表
4. 支持导出为 CSV/Excel 格式
5. 标签随新的 NPS 回复自动更新（取最新一次打分）

#### 数据字段/逻辑

**客户标签数据结构：**

| 字段 | 类型 | 说明 |
|------|------|------|
| customer_id | UUID | 客户唯一标识 |
| name | String | 客户姓名（如有采集） |
| phone | String | 手机号（如有采集） |
| wechat_id | String | 微信号（如有采集） |
| current_score | Integer | 最新 NPS 分数 |
| current_category | Enum | 当前分类 |
| score_history | Array | 历史打分记录 |
| first_survey_at | DateTime | 首次参与调研时间 |
| last_survey_at | DateTime | 最近参与调研时间 |

**业务逻辑：**
- 标签取客户最新一次 NPS 打分结果
- 如果客户参与了多次调研，保留历史记录
- 导出时包含所有可选字段
- 支持自定义标签（增强功能）

#### 验收标准
1. ✅ 客户标签根据最新打分自动更新
2. ✅ 客户列表支持按标签筛选
3. ✅ 导出功能正常（CSV 和 Excel 格式）
4. ✅ 历史打分记录可追溯

### 3.6 F6 — Detractor 告警

#### 功能描述
当收到低分（Detractor）回复时，实时通知主理人，确保第一时间了解不满客户并采取行动。

#### 用户故事
- **As a** 线下门店老板，**I want** 收到差评时立即收到微信通知，**so that** 我能在客户离开前或问题扩大前主动联系客户挽回。

#### 交互流程

**告警触发：**
1. 客户提交 NPS 打分 ≤ 6 分
2. 系统立即触发告警

**告警通知方式：**
1. **微信服务号消息**（首选）：
   - 推送模板消息到主理人微信
   - 内容："[品牌名] 收到一条低分反馈（X 分），客户留言：[留言内容]"
   - 点击跳转到该条回复详情页
2. **短信通知**（备选）：
   - 微信未关注时使用短信
3. **站内通知**：
   - 用户登录后在通知中心查看

**告警设置：**
- 用户可设置告警阈值（默认 ≤ 6 分）
- 可设置免打扰时段（默认 23:00-08:00 不推送）
- 可选择通知渠道（微信/短信/仅站内）

#### 数据字段/逻辑

| 字段 | 类型 | 说明 |
|------|------|------|
| alert_id | UUID | 告警唯一标识 |
| response_id | UUID | 关联的回复 |
| survey_id | UUID | 关联的调研 |
| score | Integer | 触发告警的分数 |
| channel | Enum | wechat / sms / in_app |
| sent_at | DateTime | 发送时间 |
| read_at | DateTime | 阅读时间 |
| status | Enum | pending / sent / read |

**业务逻辑：**
- 告警在回复提交后 ≤ 30 秒内发送
- 免打扰时段的告警延迟到时段结束后批量发送
- 同一调研的告警频率限制：每小时最多 20 条

#### 验收标准
1. ✅ 低分回复提交后 ≤ 30 秒内触发通知
2. ✅ 微信模板消息正常推送且内容准确
3. ✅ 免打扰时段设置生效
4. ✅ 告警记录可追溯

### 3.7 F7 — 留言分析

#### 功能描述
收集并展示客户文字反馈，支持关键词提取和分类展示，帮助用户快速理解客户的真实想法。

#### 用户故事
- **As a** 知识付费主理人，**I want** 看到客户留言的关键词汇总，**so that** 我能快速发现共性问题，而不是逐条阅读 100 条留言。

#### 交互流程

1. 用户进入"留言分析"页面
2. 展示所有文字留言列表（按时间倒序）
3. 顶部展示关键词词云（基于分词 + 词频统计）
4. 支持按 NPS 分类筛选（Promoter / Detractor 留言分开看）
5. 支持按关键词搜索留言
6. 点击单条留言可查看完整内容和客户信息

#### 数据字段/逻辑

**关键词提取逻辑：**
1. 对所有留言文本进行中文分词（jieba 分词引擎）
2. 过滤停用词（"的"、"了"、"是"等）
3. 统计词频，取 Top 20 关键词
4. 关键词大小按词频比例展示（词云效果）

**数据结构：**
```json
{
  "keywords": [
    { "word": "课程", "count": 45, "sentiment": "neutral" },
    { "word": "实操", "count": 32, "sentiment": "negative" },
    { "word": "老师", "count": 28, "sentiment": "positive" }
  ],
  "comments": [
    {
      "response_id": "...",
      "score": 4,
      "category": "detractor",
      "comment": "课程太理论了，希望多一些实操",
      "keywords": ["课程", "理论", "实操"],
      "submitted_at": "2026-07-05T10:30:00Z"
    }
  ]
}
```

#### 验收标准
1. ✅ 关键词提取准确（中文分词正确率 ≥ 90%）
2. ✅ 词云展示正常，点击关键词可筛选相关留言
3. ✅ 留言支持按分类和关键词筛选
4. ✅ 页面加载时间 ≤ 3 秒（1000 条留言以内）

### 4.4 盈亏平衡分析

## 5. MVP 定义

### 5.1 Phase 1：最小可行产品（目标：4 周上线）

**核心原则：** 用最小成本验证"用户是否愿意用 NPS 工具收集客户反馈"这一核心假设。

| 功能 | MVP 范围 | 实现方式 |
|------|----------|----------|
| **F1 调研创建** | ✅ 包含 | 仅支持 NPS 核心问题 + 1 个附加问题；暂不支持 logo 上传和品牌色自定义 |
| **F2 多渠道分发** | ✅ 包含 | 仅支持链接复制和二维码下载；暂不支持微信卡片和批量发送 |
| **F3 自动计分** | ✅ 包含 | 标准 NPS 计算 + 三分类；全自动化 |
| **F4 数据看板** | ✅ 包含 | 仅展示核心指标卡片 + 留言列表；暂不支持趋势图和多调研对比 |
| **F5 客户分层** | ⚠️ 半自动 | 系统自动分类，但导出功能需手动操作（暂不支持一键导出） |
| **F6 Detractor 告警** | ❌ 不含 | MVP 阶段用户手动查看看板；后续迭代加入 |
| **F7 留言分析** | ❌ 不含 | MVP 阶段仅展示原始留言列表；关键词分析后续迭代 |

### 5.2 手动/半自动 vs 自动化

| 事项 | MVP 实现方式 | 后续自动化 |
|------|-------------|-----------|
| **客户信息采集** | 调研中增加"姓名（选填）"字段 | 对接电商订单系统自动填充 |
| **Detractor 告警** | 用户每天手动查看看板 | 自动微信/短信告警 |
| **客户分层导出** | 后台手动筛选 + 复制到 Excel | 一键导出 CSV/Excel |
| **品牌定制** | 使用通用模板 | 支持 logo、品牌色、自定义模板 |
| **支付和开通** | 人工收款 + 手动开通账号 | 接入支付系统自动开通 |
| **客服支持** | 微信群 1v1 答疑 | 工单系统 + FAQ |
| **数据备份** | 每日手动备份数据库 | 自动化备份 + 灾备 |

### 5.3 MVP 技术方案概要

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端** | 微信小程序 + H5（响应式） | 小程序用于管理后台，H5 用于客户端问卷填写 |
| **后端** | Node.js / Python FastAPI | 轻量级，快速开发 |
| **数据库** | PostgreSQL | 结构化数据存储 |
| **部署** | 阿里云 ECS | 初期 2 台（应用 + 数据库） |
| **短信** | 阿里云短信 | 告警通知 |
| **二维码** | qrcode.js | 前端生成 |

### 5.4 MVP 验证指标

| 指标 | 目标 | 说明 |
|------|------|------|
| **注册转化率** | 访问 → 注册 ≥ 15% | 验证产品吸引力 |
| **调研创建率** | 注册 → 创建调研 ≥ 60% | 验证功能易用性 |
| **问卷回收率** | 发送 → 填写 ≥ 30% | 验证客户端体验 |
| **7 日留存率** | 创建调研后 7 天内再次登录 ≥ 40% | 验证产品粘性 |
| **付费转化率** | 免费试用 → 付费 ≥ 10% | 验证商业模型 |

### 5.5 MVP 上线计划

| 周次 | 里程碑 | 交付物 |
|------|--------|--------|
| W1 | 需求确认 + 技术方案 | PRD 定稿、技术架构文档 |
